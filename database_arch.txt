Database Architecture (SQLite)
Data Infrastructure Logic
The backend uses a layered database approach to ensure both development speed and code quality:

Database Engine: SQLite
We chose SQLite for its simplicity. It’s a serverless, file-based database. There’s no need to host a separate DB server; everything lives in database.db.

ORM Core: SQLAlchemy
The industry-standard Toolkit for Python. It handles the heavy lifting of connection pooling and asynchronous queries (AsyncSession).

High-Level Wrapper: SQLModel
This sits on top of SQLAlchemy and Pydantic. It allows us to use the same class for both Database Tables and Data Validation, significantly reducing code redundancy.

1. Table: users
Manages account security and user identity.

id: Primary Key (Integer)

email / username: Unique identifiers for login.

password_hash: Securely stored using passlib.

is_active: Boolean (Defaults to True).

2. Table: fitness_records
Stores physical metrics. Note: All units are Imperial.

id: Primary Key.

user_id: Link to the specific user.

height_in: Height in inches.

weight_lbs: Weight in pounds.

activity / goal: Context for the AI to provide better advice.

3. Table: fitness_reports
The output from our Mistral-Nemo/Llama3.1 models.

id: Primary Key.

report_content: The full Markdown text generated by the LLM.

data_summary: A snapshot of the user's data at the time of generation.

model_used: Tracks which AI model was used (for debugging/quality checks).

Developer Notes for the Team
Database File: The database.db file will be automatically created the first time you run python main.py.

Schema Changes: If you modify the models in src/model.py, you must delete the old database.db file so it can be re-generated with the new structure.

Tools: I recommend installing the "SQLite Viewer" extension in VS Code to view and edit the data directly.
